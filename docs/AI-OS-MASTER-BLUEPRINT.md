# SECTION 01 — EXECUTIVE PRODUCT VISION
*Why 3JN Travel OS is not a booking site — it is an AI Infrastructure Platform*

## 1. Executive Product Vision

3JN Travel OS is a purpose-engineered **AI Infrastructure Operating System** that eliminates travel complexity and minimises total trip cost through autonomous neural intelligence. It is architecturally distinct from every OTA, metasearch engine, and travel management platform currently in operation. Where legacy platforms present options, **3JN Travel OS delivers decisions.**

### 1.1 The Core Problem This Platform Solves

The global travel industry processes over **USD 9 trillion annually**. Yet the primary consumer experience has barely changed since the 1990s: fragmented interfaces, retail-priced inventory, hidden fees, disjointed post-booking management, and no continuous optimisation. The modern traveller is forced to operate as their own travel analyst, logistics coordinator, compliance officer, and risk manager simultaneously.

- **42+ browser tabs** opened for a typical international booking
- Average retail markup of **18–35%** over true wholesale rates
- **Zero post-booking intelligence** — platforms abandon users after payment
- Visa, transfer, and eSIM management remain entirely manual
- No real-time price monitoring or dynamic rebooking capability
- Risk alerts, disruption management, and FCO guidance are unintegrated

*(Engine answers, all live: `decision`/`travelCFO` deliver decisions not options; partner net rates + `deepPriceDive` attack the retail markup; `runPriceGuard`/`runDisruptionGuard` are the post-booking intelligence; VisaOS + transfers + eSIM are in-basket; the 7-layer risk feed integrates advisories.)*

### 1.2 The 3JN Travel OS Proposition

3JN Travel OS operates as a **seven-layer autonomous intelligence stack**: intent extraction, wholesale negotiation, integrity verification, loyalty injection, logistics synchronisation, continuous monitoring, and self-optimising rebooking. Every layer is powered by proprietary AI agents operating within a unified orchestration mesh.

| Layer | Intelligence Function | Commercial Outcome | Live symbol |
|---|---|---|---|
| **01 Neural Intent** | NLP extraction of 40+ travel parameters from natural language | Zero-friction search; 100% intent accuracy | `parseIntent` + `INTENT_PARAMETERS` (46 registered) |
| **02 Wholesale Engine** | Direct GDS + wholesaler API negotiation bypass | 18–35% cost reduction at booking | `partners.js` net rates + `negotiation` |
| **03 Integrity Shield** | 50-point verification against supplier database | 100% verified inventory only | `INTEGRITY_CHECKS` (50) + reliability floor |
| **04 Loyalty Injection** | Membership tier discount auto-application | Additional 2–15% member savings | `LOYALTY_TIERS` in `priceBreakdown` |
| **05 Universal Console** | Centralised visa, transfer, eSIM, and document management | Single-pane journey command centre | Console view + VisaOS + esims |
| **06 Continuous Monitor** | 24/7 price monitoring and disruption detection | Post-booking savings + zero surprises | `runPriceGuard` (Neural Price Guard) |
| **07 Autonomous Reoptimise** | AI-triggered rebooking on price drops or disruptions | Lifetime trip cost reduction | `runDisruptionGuard` (cost-neutral rebooking) |

3JN Travel OS occupies a **category of one**. It is not competing with Booking.com, Expedia, or Skyscanner at the search layer. It is competing with the entire concept of manual travel management. The closest analogy is a **fully autonomous CFO for every journey** — always negotiating, always monitoring, always optimising.

### 0.3 Gap Register — findings → proven-pattern remedies

Severity: **P0** ship-blocker for real money/PII · **P1** scale/commercial · **P2** hardening.

| # | Lens | Finding (current state) | Proven-pattern remedy | Ref pattern |
|---|---|---|---|---|
| G1 | Technical | State lives in `store.js:db` (in-memory `Map`s) — lost on restart; no ACID; single process. | Promote to **Postgres 16** (OLTP) + **Firestore** for realtime read models; keep the `store.js` interface as a repository port. | Stripe/Uber OLTP |
| G2 | Security | Money & PII flows exist (bookings, deposits, BitriPay links, KYC-adjacent VisaOS) with no encryption-at-rest, no field tokenisation, no PCI scope isolation. | **Envelope encryption** (KMS + per-field DEK), tokenise PAN/passport via vault, never touch raw PAN (hosted fields). | Stripe PCI, Vault |
| G3 | Security | AuthN is Firebase + a local `/api/login`; no MFA, no device fingerprint, no risk-based step-up; API keys stored plaintext in `store.js:apiKeys`. | **Zero-Trust identity**: MFA/TOTP+WebAuthn, device fingerprint, risk-based auth; API keys stored as **salted hash + prefix**, shown once. | CrowdStrike/Okta |
| G4 | Scalability | Synchronous request path runs intent→scan→package inline; supplier scan is CPU-bound; no queue, no cache tier. | **Event-driven** core (Kafka/PubSub) + **Redis** cache + worker pool; search becomes an async saga with cached read models. | Uber dispatch |
| G5 | AI | Gateway `callProvider` is a stub; no retries, no eval harness, no prompt versioning, no guardrails, no agent memory store. | Wire providers with **timeout+retry+circuit-breaker**, **eval/regression suite**, **prompt registry**, **guardrail filter**, **pgvector agent memory**. | OpenAI/Anthropic prod |
| G6 | AI | Agents (`agents.js`, `visaos.js`) are single-shot deterministic reports; no orchestration, no inter-agent messaging, no human-in-the-loop queue. | **Orchestrator-worker** multi-agent runtime with a shared blackboard, tool registry, and escalation queue. | Anthropic agents |

## 1. Executive Product Vision

**What it is.** 3JN Travel OS is an **AI Infrastructure Operating System for global travel commerce** — not a booking site. It ingests a one-sentence intent, understands it, funds the work only when profitable (ACPE), scans verified global supply, assembles the *cheapest reliable* package across every vertical, prices it transparently, offers deposit + interest-free instalments, keeps guarding the price after booking, and exposes the whole engine as a **white-label + API + payments platform** other businesses build on.

**The problem it solves.** Travel search is fragmented (flights here, hotels there, visa somewhere else, roaming forgotten), opaque (real price hidden behind fees), and static (no one watches your booking after you pay). Businesses lose margin to OTAs; the diaspora and price-sensitive travellers are structurally underserved by instalment-free, visa-blind incumbents.

**Why it is different.** Three defensible primitives already in code:
1. **ACPE profit-gate** (`revenue.js`) — the OS refuses to spend AI/compute unless `revenue ≥ cost × 10`. Unit economics are enforced by the runtime, not hoped for. No incumbent does this.
2. **VisaOS decision engine** (`visaos.js`) — approval-probability *before* booking turns a booking blocker into a conversion lever.
3. **Neural Price Guard** (`monitor.js`) — post-purchase value creation (rebook/refund) that compounds trust and the savings-share revenue stream.

**Why the market needs it & why it can dominate.** Every travel business needs the same four things — supply access, transparent pricing, financing, and post-sale price protection — yet builds them badly and repeatedly. 3JN packages them as infrastructure (API + white-label + BitriPay), so the OS grows as a **platform tax on other people's travel commerce**, not just direct bookings. That is the Stripe/AWS wedge applied to travel.

### 5C. Agent Runtime (how agents actually run)

- **Pattern:** Orchestrator-Worker with a shared **blackboard** (Redis) + durable **task ledger** (Postgres). The orchestrator decomposes a goal, dispatches to workers (tools/sub-agents), collects results, and writes decisions to the audit chain.
- **Tool registry:** each agent may call only whitelisted tools (internal APIs, connectors, SQL read models) — enforced by scope.
- **Memory:** short-term (conversation, Redis TTL) + long-term (**pgvector** embeddings keyed by principal) + episodic (past decisions in the ledger).
- **Human-in-the-loop:** an escalation queue; VisaOS "Human Review" band (`visaos.js`) is the reference implementation.
- **Determinism preserved:** every LLM tool has a deterministic local fallback (the current `agents.js`/`visaos.js`/`intent.js` logic), so an agent never hard-fails — mirroring `ai-gateway.js:run`.

## 7. BitriPay Payment Gateway API Door

A dedicated, plugin-ready gateway so merchants, partners, and external platforms install BitriPay in minutes. Extends existing `/api/bitripay/*` + `store.js:paymentLinks`.

**Merchant Integration Portal**
- **API key management:** publishable (`pk_`) + secret (`sk_`) keys, **sandbox** (`_test_`) and **live** (`_live_`) modes; secret stored as `argon2` hash + last-4 shown once; rotate/revoke; scopes (`payments:charge`, `links:write`, `settlement:read`).
- **Webhook management:** register endpoints, **HMAC-SHA256 signed** (`BitriPay-Signature: t=...,v1=...`), timestamped, replay-protected (300s window), auto-retry with exponential backoff + dead-letter, delivery log.
- **Environments:** sandbox (fake settlement, instant results) mirrors live 1:1.

**Payment services:** QR payments, wallet payments, card (hosted fields — PAN never touches 3JN servers, G2), bank transfers, mobile money. Each returns a `PaymentIntent`-style object with idempotent creation.

**Developer Centre:** SDKs (JS/TS, Python, PHP/WordPress plugin, mobile), OpenAPI 3.1 spec, Postman collection, interactive docs, test cards/numbers, request signing helper.

**Revenue management:** fee management (per-merchant pricing), **commission split** (agent/partner shares — the Rayna-style net-rate economics), **settlement engine** (T+N payout batches, reconciliation, statements), refunds/disputes/chargebacks with evidence submission.

**Core objects & endpoints:**

```
POST /api/bitripay/links                 create payment link / QR      (idempotent)
GET  /api/bitripay/links                 list links (tenant-scoped)
POST /api/bitripay/links/:id/settle      settle → ledger + payout
POST /api/bitripay/charges               create charge (hosted fields token)
POST /api/bitripay/refunds               refund (partial/full)
POST /api/bitripay/disputes/:id/evidence submit dispute evidence
POST /api/bitripay/webhooks              register endpoint
GET  /api/bitripay/settlements           payout batches + reconciliation
```

**Webhook events:** `payment.succeeded`, `payment.failed`, `link.paid`, `settlement.completed`, `refund.processed`, `dispute.opened`, `dispute.won/lost`, `payout.paid`.

**Plugin-ready logic:** a single `BitriPay.init({ publishableKey, mode })` client + server SDK; WordPress/WooCommerce, Shopify app, and REST modes share one signing/verification contract.

### 9.2 Component mandates

- **Observability:** OpenTelemetry traces (Tempo) + metrics (Prometheus/Grafana) + logs (Loki); RED/USE dashboards; SLOs + error-budget alerts.
- **Error handling:** the platform's signature discipline — **every handler returns clean JSON, never an empty object** (preserved from `server.js`); typed error codes (§11); circuit-breakers; graceful degradation to deterministic fallback.
- **Scalability:** stateless services + HPA on k8s; async search saga; cache tier; read replicas; partitioned event log.

## 11. API Specification (REST + webhooks)

**Conventions:** base `/api/v1`; auth via `Authorization: Bearer <oidc>` (users) or `Authorization: Bearer sk_live_...` (server keys); **idempotency** via `Idempotency-Key` header on all POSTs that move money; JSON only (never empty object); rate limits per tier.

**Auth & errors:**
```
401 unauthorized · 403 forbidden · 404 not_found · 409 conflict/duplicate
422 validation_error · 429 rate_limited · 402 payment_required (ACPE downgrade)
5xx internal → { error: { code, message, request_id } }  // always JSON
```

**Representative endpoints (existing → productionised):**
```
POST /api/v1/plan
  headers: Idempotency-Key, X-Currency (optional)
  body:   { "text": "Dubai family August 7 nights flights hotel visa instalments" }
  200:    { quoteId, options:[{tier,totalUsd,components,recommended}], funding, acuCharged }
  402:    { downgradeTo:"free", requirement:{ acuNeeded, orDepositGBP, orSubscription } }  // ACPE

POST /api/v1/quotes/:id/book
  headers: Idempotency-Key
  body:   { paymentMethod, depositUsd }
  201:    { bookingId, status:"confirmed", instalmentPlan:[...], ledger:[...] }
  409:    { error:{ code:"duplicate", bookingId } }               // idempotent replay

POST /api/v1/bookings/:id/pay            → instalment charge (idempotent)
POST /api/v1/bookings/:id/price-guard    → { action:"rebook|hold", refundUsd }
POST /api/v1/visaos/assess               → { totalScore, decision, dimensions, confidence }
GET  /api/v1/visaos/probability?...      → approval-probability badge
POST /api/v1/bitripay/links              → { id, qrUrl, status } (idempotent)
POST /api/v1/bitripay/links/:id/settle   → { settlementId, net, split }
POST /api/v1/merchant/keys               → { id, prefix, secretOnce }  // shown once (G3)
GET  /api/v1/white-label/payout?volume=  → { commissionUsd, partnerNetUsd, platformShareUsd }
GET  /api/v1/admin/overview              → platform KPIs (admin scope)
POST /api/v1/agents/:agent/run           → { runId, output(STANDARD_OUTPUT_FORMAT), acu, mode }
```

**Rate limits (per key/user/min):** Nomad 30 · Family 60 · Executive 120 · Elite 300 · Merchant/Partner by contract; burst via token bucket at the edge.

**Webhooks (signed HMAC-SHA256, retried, replay-protected):**
```

visaos.decision · kyc.completed · agent.escalated
```

## 13. Security, Compliance & Risk

- **GDPR / privacy:** consent ledger (`consents`), DSAR export/delete workflow, purpose limitation, data-residency routing, retention schedules.
- **KYC/KYB:** Onboarding + Compliance agents via Persona/Sumsub; status on `users.kyc_status`.
- **AML:** sanctions/PEP screening at onboarding + tx thresholds; SAR filing workflow; MLRO escalation.
- **PCI-DSS:** hosted card fields (SAQ-A scope), no raw PAN, tokenisation, network segmentation for payment services.
- **Audit:** hash-chained append-only `audit` (prev_hash→hash), exportable, tamper-evident.
- **Fraud:** transaction + behaviour + device scoring (Fraud agent); VisaOS scoring model reused for identity risk.
- **RBAC/ABAC:** OPA policy-as-code; least privilege; `allAccess`/break-glass require dual-control + full audit.
- **Encryption:** at rest (KMS envelope, per-field DEK), in transit (TLS 1.3, mTLS internal), in use (field-level/confidential compute).
- **API keys:** salted-hash storage, prefix display, scopes, rotation, once-only reveal.
- **Webhook signing:** HMAC-SHA256, timestamped, replay-protected.
- **Transaction monitoring:** real-time rules + ML anomaly detection; velocity checks.
- **Admin action tracking:** every admin/agent action recorded with actor, before/after, request_id.
- **Data classification:** public / internal / confidential / restricted (PII, PAN, passport) → drives encryption + access.

## 16. Competitive Advantage

- **Runtime-enforced margins.** ACPE (`revenue.js`) makes profitability a property of the system, not a spreadsheet — no competitor gates compute on `revenue ≥ cost × 10`.
- **Decision intelligence pre-purchase.** VisaOS turns the industry's #1 booking blocker into a conversion and data moat.

## Appendix B — Agent Workflow Example (funded Deep Search)
1. `search.requested` → **Revenue/ACPE** runs `costProtectionGate({tier:'deep', user, expectedBookingUSD})`.
2. If unfunded → emit `acpe.decided{downgradeTo:'free'}`, return `402` with requirement (never blocks the user).
3. If funded → **Orchestrator** dispatches: Intent → Supplier scan → Risk (`riskFeed`) → VisaOS probability → Packager → Pricing.
4. Each worker writes to the blackboard; orchestrator assembles tiered options + recommendation.
5. `quote.created`; ACU debited via gateway metering; audit row appended (hash-chained).
6. On book → **Payment** posts double-entry ledger idempotently; **Price Guard** schedules post-sale re-shop.
